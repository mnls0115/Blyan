How Blyan Works

From user query to AI response - powered by blockchain consensus

Input & Response Flow

When you send a message, GPU nodes collaborate in a pipeline to generate the response. Node 1 handles embedding and first layers, Node 2 processes middle layers, Node 3 handles final layers and output generation. The coordinator orchestrates this pipeline parallelism, routing hidden states between nodes efficiently. Each node loads only its assigned layer weights from the blockchain.

  • Pipeline setup: Coordinator assigns layer ranges to nodes based on GPU memory capacity
  • Sequential processing: Hidden states flow through pipeline stages from embedding to output
  • Zero-copy streaming: Tensor data streams between nodes without CPU transfers
🧠
2

Proof-of-Learning (PoL) & Model Updates

Through our Proof-of-Learning (PoL) consensus mechanism, nodes validate model improvements by demonstrating measurable quality gains. When idle, nodes automatically engage in PoL training to improve their assigned layers. Contributors evolve individual layer weights through delta blocks and migration blocks that track parameter changes. During high inference demand, PoL accelerates as the system accumulates rewards that can be distributed to contributing nodes.

  • Proof-of-Learning validation: Nodes must prove their training improved model quality to earn rewards
  • Idle time PoL training: Nodes train their assigned layers when not serving inference requests
  • Demand-driven evolution: High inference activity generates more rewards, incentivizing rapid improvements
  • Layer evolution: New layer versions branch from existing ones, validated through PoL consensus
💎
3

Participation & Rewards

Nodes join the network by registering their GPU capabilities and assigned layer ranges. When active, they serve inference requests as part of the pipeline - Node 1 handles input layers, Node 2 processes middle layers, Node 3 generates output. During quiet periods, nodes switch to idle time PoL training, improving their assigned layers. Rewards are automatically distributed based on actual pipeline participation and layer performance.

  • Node registration: Join with GPU specs and receive appropriate layer assignments
  • Pipeline participation: Process assigned layer computations as part of distributed inference
  • Idle time PoL training: Train and improve assigned layers during idle time
  • Performance-based rewards: Distribution based on pipeline participation and layer quality
📚
4

Data Pool & Training Data Flow

The Blyan Data Pool ensures that training data is transparent, deduplicated, and reusable across the network. User uploads (JSON, CSV, PDF, TXT) and curated internet datasets (Wikipedia, C4, BookCorpus, etc.) are ingested through adapters, normalized, and chunked into token-level segments. Each piece of data is verified, scored, and stored as a content-addressed block with Merkle proofs for auditability.

  • Ingestion & Validation: Data passes through format adapters, PII redaction, license checks, and deduplication (BLAKE3 + MinHash/SimHash). Quality scoring classifies chunks into Gold, Silver, or Bronze tiers.
  • Transparent Storage: Verified chunks are stored in a content-addressed system, with metadata logged to Chain D for integrity. The catalog is dual-layered: a private index for operators and a public version for transparency.
  • No-Reuse Guarantee: Each expert tracks its used data via efficient Cuckoo filters, ensuring new epochs always receive fresh, unseen data.
  • Fair Distribution: The scheduler balances data across GPU nodes using consistent hashing, guaranteeing coverage across experts while avoiding duplication.
  • Right-to-be-Forgotten: Users can request deletion at any time, triggering a full GDPR-compliant removal across catalogs, storage, and usage ledgers.

By combining open internet data and user-contributed datasets into a single verifiable pool, Blyan ensures that every training step is auditable, fair, and reproducible, while contributors retain control over their data.

Key Technologies

The building blocks that make decentralized AI possible

⚡

TensorBlock Format

Efficient storage format with compact header and contiguous tensor data for instant loading. Supports multiple precision levels with quantization. Integrity verification without loading full tensors.

🔗

Dual-Chain Architecture

Chain A stores model architecture specifications, Chain B stores layer weights. Dependency tracking enables parallel layer evolution branches that merge intelligently.

🛡️

Evolution Block System

Blocks track parent relationships, version numbers, evolution types, and dimensional changes. Compatible versions merge automatically while incompatible branches fork.

🎯

Layer Pipeline & Caching

Coordinator assigns layer ranges to GPU nodes based on memory capacity and performance. Frequently accessed layers stay cached in GPU memory with usage statistics tracked for rewards.

🔄

Distributed Pipeline Network

Nodes register their GPU capabilities and assigned layer ranges. The coordinator orchestrates pipeline parallelism, routing hidden states between nodes with automatic failover.

📊

Reward Distribution System

The system tracks contributions and distributes rewards based on actual usage, quality metrics, and network demand. Maintains economic balance through controlled token distribution.
